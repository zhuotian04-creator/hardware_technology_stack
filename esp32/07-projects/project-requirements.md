# 项目需求

项目需求是综合项目的起点。先把需求写清楚，后面才知道该选哪些模块、哪些传感器、哪些通信方式。

## 1. 需求应该包含什么

1. 项目目标
2. 用户和使用场景
3. 输入、处理和输出
4. 功能列表和优先级
5. 供电方式
6. 体积和结构限制
7. 通信方式
8. 异常与安全状态
9. 可验证的验收条件

“实现环境监测”还不是可以直接开发的需求。“每 2 秒读取一次温湿度，并在网页中显示；温度超过 30°C 时点亮报警灯”才可以被实现和检查。

把“明确不做”写出来很重要，它能阻止项目在临近交付时不断增加功能。

## 3. 示例：环境监测终端

| 项目 | 内容 |
| --- | --- |
| 目标 | 在局域网中查看房间温湿度，并在过热时报警 |
| 输入 | BME280 温度和湿度 |
| 处理 | 每 2 秒采样；温度大于 30°C 报警，小于 28°C 恢复 |
| 输出 | OLED 显示、板载 LED、Web 页面 |
| 通信 | STA 模式连接 Wi-Fi，浏览器访问设备 IP |
| 供电 | USB 5V |
| 断网行为 | 继续本地采样和报警，每 10 秒尝试重连 |
| 传感器故障 | 页面显示故障状态，不使用上一次数据触发新动作 |

对应的验收条件可以写成：

1. 上电 10 秒内完成初始化并在串口打印状态。
2. 连续运行时，每 2 秒更新一次测量值。
3. 温度跨过设定阈值后 3 秒内改变报警状态。
4. 手机在同一局域网中能打开页面并看到最新数据。
5. 关闭热点再恢复后，设备能自动重新连接。
6. 断电重启后不需要重新上传程序。

## 4. 开工前的方案表

根据需求列出每个模块，确认它们能同时工作：

| 功能 | 模块或方案 | 接口 | 引脚/地址 | 供电 | 已单测 |
| --- | --- | --- | --- | --- | --- |
| 温湿度 | BME280 | I2C | `0x76` | 3.3V | 否 |
| 显示 | OLED | I2C | `0x3C` | 3.3V | 否 |
| 报警 | 板载 LED | GPIO | `GPIO 2` | 板载 | 否 |

表中的型号、地址和引脚只是示例，必须按实际硬件修改。

## 5. 学习目标

1. 先把问题定义清楚。
2. 再选硬件和软件方案。
3. 最后再开始写代码。
4. 能把模糊想法改写成可验证条件。
5. 能提前定义故障和断网时的设备行为。

# 调试方法

调试能力是训练营里必须单独训练的能力。真正会做项目的人，不是从不出错，而是知道怎么快速定位问题。

## 1. 固定调试顺序

1. 写清楚“期望现象”和“实际现象”。
2. 判断问题发生在供电、接线、单模块、程序流程还是网络。
3. 把完整项目缩小到能复现问题的最小版本。
4. 每次只改变一个条件，并记录结果。
5. 修复后恢复完整项目，再做一次断电重启验证。

不要同时换线、换引脚、改代码和改库版本，否则即使恢复正常，也很难知道真正原因。

## 2. 常见调试手段

- 串口打印
- 逐步注释
- 单模块验证
- 观察硬件反应
- 检查供电和接线
- 用万用表测量电压和通断
- 对照最小示例和模块资料

## 3. 让日志有意义

日志应包含模块、状态和关键数值：

```cpp
Serial.printf("[SENSOR] raw=%d\n", rawValue);
Serial.printf("[WIFI] status=%d, rssi=%d\n", WiFi.status(), WiFi.RSSI());
Serial.printf("[STATE] %d -> %d\n", oldState, newState);
```

不要只打印 `error`。更有用的写法是说明哪个步骤失败、返回值是什么、设备接下来准备做什么。高频循环中的日志要限速，否则串口输出本身会影响时序。

## 4. 常见现象

| 现象 | 优先检查 |
| --- | --- |
| 无法上传 | 数据线、端口、开发板型号、BOOT 按键、串口占用 |
| 不断重启 | 电源压降、短路、看门狗、启动日志中的复位原因 |
| I2C 找不到设备 | 地址、SDA/SCL、供电、共地、上拉电阻 |
| 接入 Wi-Fi 后模拟量异常 | 是否使用了经典 ESP32 的 ADC2 引脚 |
| 电机动作时系统重启 | 电源容量、共地、驱动器、反向电动势和干扰 |
| 输出动作相反 | 模块是否低电平有效、引脚初始状态 |
| 程序运行一段时间后卡住 | 阻塞循环、重连逻辑、内存增长、任务执行时间 |

## 5. 单模块隔离法

如果综合项目出现问题，按这个顺序恢复功能：

1. 只保留串口和一个 LED，确认程序能稳定运行。
2. 一次接入一个传感器，并打印原始值。
3. 一次接入一个执行器，先用固定命令控制。
4. 最后加入 Wi-Fi、Web 或 MQTT。
5. 每加入一项就运行足够长时间，确认问题在哪一步出现。

## 6. 调试记录

每次问题至少记录：日期、硬件连接、代码版本、复现步骤、串口日志、尝试过的修改和最终原因。照片和接线图通常比一句“接线没问题”更有价值。

## 7. 学习目标

1. 先缩小问题范围。
2. 先看串口输出。
3. 能区分硬件、软件和网络问题。
4. 能用最小程序复现问题。
5. 能留下别人可以继续排查的记录。
